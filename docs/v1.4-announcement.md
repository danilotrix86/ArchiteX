# ArchiteX 1.4: Azure support, auto-detected

> **TL;DR** — ArchiteX is a free, open-source GitHub Action that reads the Terraform diff in a PR and posts a deterministic architectural review back as a sticky comment. v1.4 adds first-class **Azure** support: 12 new `azurerm_*` resources, 3 Azure-specific risk rules, and **automatic provider detection** — drop your Terraform in and the tool figures out whether each resource is `aws_*` or `azurerm_*` and applies the right rules. AWS-only repos see zero behavioral change. Try it: [github.com/danilotrix86/ArchiteX](https://github.com/danilotrix86/ArchiteX).

## What's new

- **12 new `azurerm_*` resource types** covering the canonical Azure 3-tier scope:
  - **Network:** `azurerm_virtual_network`, `azurerm_subnet`, `azurerm_public_ip`, `azurerm_network_interface`
  - **Access control:** `azurerm_network_security_group`, `azurerm_network_security_rule`
  - **Compute:** `azurerm_linux_virtual_machine`, `azurerm_windows_virtual_machine`
  - **Entry points:** `azurerm_lb`
  - **Storage:** `azurerm_storage_account`
  - **Data:** `azurerm_mssql_server`, `azurerm_mssql_database`

  **57 resources total** (45 AWS + 12 Azure).

- **3 new Azure-specific risk rules** — same per-resource signal pattern as the AWS rules, same `phase6CapPerRule` cap, same library-mode `isConditionalNode` guard:
  - `nsg_allow_all_ingress` (3.5) — an NSG rule allows `*` / `0.0.0.0/0` inbound with literal `Allow`.
  - `storage_account_public` (4.0) — a storage account is added with `public_network_access_enabled = true` or `allow_nested_items_to_be_public = true`.
  - `mssql_database_public` (3.5) — an MSSQL server is added with `public_network_access_enabled = true`.

  **21 deterministic rules total.**

- **Auto-detection.** ArchiteX inspects the literal `resource "<type>" "<name>"` in every `.tf` file, looks the type up in a single registry, and applies whichever rules are appropriate. The PR comment now opens with a deterministic provider banner:

  ```
  _Detected providers: aws, azurerm — 27 resources analyzed._
  ```

  Single-provider repos see a single name. Mixed AWS + Azure repos see both. Empty deltas show no banner at all. No config required.

- **Cross-provider rules apply automatically.** The five abstract-type rules (`new_entry_point`, `new_data_resource`, `public_exposure_introduced`, `potential_data_exposure`, `resource_removed`) and the three baseline `first_time_*` rules already key off **abstract type**, not provider. So they fire on Azure as soon as the registry knows the new types — no rewrite needed.

- **AWS bit-identical promise.** Every existing AWS regression remains green. The three new Azure rules return `nil` immediately for any non-`azurerm_*` node — locked in by an explicit test. AWS-only repos can keep tracking the floating `v1` tag and see exactly the same scores, summaries, and Mermaid diagrams as v1.3.

- **New example: `examples/07-azure-public-lb/`** — the canonical Azure "public LB + open NSG" anti-pattern, wired into the examples-gallery selftest. The cross-provider equivalent of [example 01 (public ALB)](../examples/01-public-alb/).

- **`azurerm_resource_group` is intentionally excluded.** It is purely organizational, has no architectural-review value, and including it would clutter every Azure diagram with an inert root node. References to it from other resources simply do not produce edges (same warn-and-skip behavior as `var.*` / `data.*`).

## Why "auto-detect" instead of a `provider:` config

The user shouldn't have to tell ArchiteX what's in their own `.tf` files. Mixed-provider repos exist. Renaming-providers-as-you-migrate-to-Azure repos exist. The simplest, least-surprising design is: **the resource block declares its own provider** — `aws_lb`, `azurerm_lb` — and the tool reads it.

There is no `provider:` field in `.architex.yml`. There is no flag. Drop your Terraform in. It works.

## How it scales

The v1.4 architecture is the same one used since v1.0; v1.4 just exercises it with a second provider for the first time. Every future provider follows the same five-step pattern:

1. Register the new resource types in `models.SupportedResources`.
2. Map them to existing abstract roles (`compute`, `data`, `entry_point`, …) in `models.AbstractionMap`.
3. Add provider-specific edge pairs to `graph.edgeTypeMap` and any literal-attribute promotions to `graph.deriveAttributes`.
4. Add provider-specific rules in `risk/rules_<provider>.go` only where the existing cross-provider rules don't already cover the case.
5. Wire fixtures + examples + tests.

That's it. No core rewrite, no parser dance, no risk-engine overhaul. Azure tranche-1 (Application Gateway, AKS, Cosmos DB, Key Vault, Function App, Front Door) and GCP will land the same way.

## 30-second quickstart

Add this file to your Terraform repo at `.github/workflows/architex.yml`:

```yaml
name: ArchiteX
on: pull_request
permissions:
  contents: read
  pull-requests: write
jobs:
  architex:
    runs-on: ubuntu-latest
    steps:
      - uses: danilotrix86/ArchiteX@v1.4.0
        with:
          terraform-dir: infra
```

Open a PR. ArchiteX posts a sticky comment with the score, the diagram, the reasons, and (if your repo touches both clouds) a `Detected providers: aws, azurerm` banner. Add `mode: blocking` to fail the build on high-risk changes.

## Versioning

v1.4 is **purely additive**. AWS coverage and behavior are unchanged.

- Want only the AWS path you've already validated? Pin to `@v1.3.1`.
- Want the floating v1 tag with the latest stable v1.x? Pin to `@v1`. AWS results stay bit-identical.
- Want Azure today? Pin to `@v1.4.0`.

`v2.0` is reserved for a more substantial shift (likely a non-Terraform IaC language or a rework of the public Go API surface). Until then, v1.x stays additive.

## Trust model (unchanged)

- Analysis runs **locally on the runner**. No `.tf` source is transmitted anywhere.
- The only network call is the PR comment, with the same `GITHUB_TOKEN` your other workflow steps already have.
- An audit bundle (`summary.md`, `score.json`, `egress.json`, `report.html`, `manifest.json` with SHA-256 checksums) is uploaded every run.

No SaaS. No telemetry. No paid tier. No account.
