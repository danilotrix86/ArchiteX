# ArchiteX 1.3: an architecture firewall for your Terraform Pull Requests

> **TL;DR** — ArchiteX is a free, open-source GitHub Action that reads the Terraform diff in a PR and posts a deterministic architectural review back as a sticky comment. v1.3 ships today with EKS + Auto Scaling + RDS-groups support (45 AWS resources), 3 new risk rules (18 total), and a brand-new **library mode** for module-author repos. Try it: [github.com/danilotrix86/ArchiteX](https://github.com/danilotrix86/ArchiteX).

---

## The problem

PR reviews of Terraform diffs are a slog.

You scroll through 600 lines of HCL, mentally tracking which `count` block depends on which `var`, whether that new `cidr_blocks = ["0.0.0.0/0"]` is on the public subnet or the private one, and whether someone just attached `AdministratorAccess` to a service role for "convenience".

Existing tools mostly tell you _which line_ has a problem (`tfsec`, `Checkov`, etc). That catches the easy stuff. It doesn't catch:

- **Architectural drift** — a brand-new entry point appears in a repo that has never had one.
- **Conditional gates** — a resource gated behind `count = var.create ? 1 : 0` that nobody noticed.
- **Topology changes** — a security group flipped from `10.0.0.0/16` to `0.0.0.0/0` while three unrelated resources moved around it.

ArchiteX answers the question a reviewer actually needs answered: **what changed in the architecture, and why should I care?**

## What ArchiteX does

A strict four-stage pipeline:

```
base/*.tf  ─┐
            ├─> 1. Fact-Finder  ──>  2. Delta Engine  ──>  3. Risk & Policy  ──>  4. Interpreter  ──>  Sticky PR comment
head/*.tf  ─┘    HCL parse           added · removed       18 deterministic        Mermaid · MD          + audit bundle
                 graph build         · changed             rules                   · HTML                (SHA-256 manifest)
```

Every stage is **fully deterministic**. The Interpreter is template-based — no inference, no model calls, no network. The same input produces byte-identical output across runs.

## What's new in v1.3

- **Tranche-3 AWS coverage** — EKS family (`aws_eks_cluster`, node groups, addons, fargate profiles, identity provider configs), Auto Scaling family (`aws_launch_template`, `aws_autoscaling_group`, `aws_autoscaling_policy`), and RDS auxiliary groups (`aws_db_subnet_group`, `aws_db_parameter_group`, `aws_db_option_group`). **45 AWS resource types total.**
- **3 new risk rules** — `eks_public_endpoint` (3.5), `eks_no_logging` (1.5), `asg_unrestricted_scaling` (1.0). **18 deterministic rules total.**
- **Library mode** — opt-in parser mode for module-author repos. Resources gated behind `count = var.create ? 1 : 0` (and `length(var.X) > 0 ? 1 : 0`) are now materialized as **conditional phantoms** marked with `?` in the diagram. Risk rules refuse to score them, so phantoms never produce false positives.
- **Examples gallery** — six reviewer-grade scenarios with full base/head snapshots, each one a CI regression check that fails the build if the rendered output silently changes.
- **GitHub Pages site** — landing page + a fresh sample `report.html` re-rendered on every push to `main`, so anyone can see what the report looks like without installing anything.
- **Prebuilt binaries** — linux/darwin/windows on amd64+arm64, attached to every release via goreleaser.

## 30-second quickstart

Add this file to your Terraform repo at `.github/workflows/architex.yml`:

```yaml
name: ArchiteX
on: pull_request
permissions:
  contents: read
  pull-requests: write
jobs:
  architex:
    runs-on: ubuntu-latest
    steps:
      - uses: danilotrix86/ArchiteX@v1.3.0
        with:
          terraform-dir: infra
```

That's it. Open a PR. ArchiteX posts a sticky comment with the score, the diagram, and the reasons. Add `mode: blocking` to fail the build on high-risk changes.

## What it looks like

A developer adds a public ALB and opens a security group from `10.0.0.0/16` to `0.0.0.0/0`. ArchiteX posts:

> **❌ Risk Level: HIGH (9.0/10)**
>
> Status: `fail` · Severity: `high`
>
> **Plain-English Summary**
>
> This change introduces significant architectural risk. 1 node was added, 1 node was changed, and 2 edges were added. A new entry point (`aws_lb.web`) was added, increasing the public attack surface. A security-related resource (`aws_security_group.web`) was modified, which may affect access control policies.
>
> **Suggested Review Focus**
> - Exposure (4.0): A resource changed from private to public — confirm the CIDR allow-list was intended.
> - Exposure (3.0): A new entry point was added — confirm authentication and TLS are configured.
> - Data exposure (2.0): Public exposure was introduced alongside an access-control change.
>
> *(plus a Mermaid diagram showing the changed nodes and one layer of context)*

Live sample: [danilotrix86.github.io/ArchiteX/report.html](https://danilotrix86.github.io/ArchiteX/report.html).

## Trust model

- **Analysis runs locally on the runner.** No `.tf` source is transmitted anywhere.
- **The only network call is the PR comment.** Same `GITHUB_TOKEN` your other workflow steps already have.
- **Audit bundle uploaded every run.** `summary.md`, `score.json`, `egress.json`, `report.html`, `manifest.json` with SHA-256 checksums.

No SaaS. No telemetry. No paid tier. No account.

## What I'd love feedback on

1. **Coverage gaps.** What AWS resource type or Terraform pattern broke ArchiteX in your repo? Open an Issue with the smallest reproducer you can — that's the single highest-value contribution.
2. **False positives.** If a rule fired when it shouldn't have, file a bug. Trust is the entire product.
3. **Multi-provider.** Azure / GCP next on the roadmap. If you'd help shape the design, leave a comment in the [v2 discussion](https://github.com/danilotrix86/ArchiteX/discussions).
