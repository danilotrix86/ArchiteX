# v1.2 Real-World Validation — Findings

This document captures what we learned by running ArchiteX v1.2 against
the most-downloaded `terraform-aws-modules/*` repos. It exists to:

1. Document the **stability bar** met before any public promotion.
2. Track concrete **tranche-3 expansion candidates** (resources/rules
   that real codebases need but v1.2 doesn't yet cover).
3. Set **realistic user expectations** about what ArchiteX understands
   today vs. what it intentionally skips.

The harness that produces this evidence is
[`scripts/validate-real-world.ps1`](../scripts/validate-real-world.ps1);
re-run it any time and re-triage.

---

## Run 1 — 10-repo smoke set (2026-04-20)

Smoke set: `terraform-aws-vpc`, `terraform-aws-eks`, `terraform-aws-rds`,
`terraform-aws-s3-bucket`, `terraform-aws-iam`, `terraform-aws-lambda`,
`terraform-aws-security-group`, `terraform-aws-alb`, `terraform-aws-ecs`,
`terraform-aws-cloudfront`. All shallow-cloned from `main`.

| Bar | Result |
| --- | --- |
| Parser panics | **0 / 10** |
| Non-zero exits | **0 / 10** |
| JSON-malformed graphs | **0 / 10** |
| Median runtime | **~1 s/repo** |

ArchiteX did not crash, hang, or produce malformed output on any of the
10 most-downloaded community modules. That's the v1.2 stability bar
this gate exists to enforce, and it's met.

### What got skipped (and why)

Every "library-style" module in the smoke set wraps each resource in
`count = var.create ? 1 : 0` (or a similar variable-driven `count` /
`for_each`). v1.2's PR1 expansion is **deliberately conservative**: it
only materializes a `count` / `for_each` when the controlling
expression is a literal we can resolve at parse time. Anything that
depends on a variable, function call, or another resource's output is
warn-and-skipped rather than guessed at.

**Implication for users**: ArchiteX produces an empty graph from a
*library* repo whose every resource is gated by `var.create`. That is
the correct, conservative behavior — but a user pointing ArchiteX at
such a library will see "0 nodes / 0 edges" and reasonably ask "why?".
The high-confidence answer in v1.2 is: this tool is designed to review
**consumers** of those libraries (root modules with concrete inputs),
not the libraries themselves.

### Tranche-3 expansion candidates (from real warnings)

The `unsupported resource type "..."` warnings are the highest-signal
output of this run — each one is a real type used by real customers
that v1.2 cannot model. From the smoke-set logs, the top recurring
unsupported types are:

- **EKS** — `aws_eks_cluster`, `aws_eks_node_group`,
  `aws_eks_addon`, `aws_eks_identity_provider_config`,
  `aws_eks_fargate_profile`. **Highest priority** — every modern
  Kubernetes-on-AWS shop has these.
- **RDS / database** — `aws_db_subnet_group`,
  `aws_db_parameter_group`, `aws_db_option_group`,
  `aws_db_instance` (already partially supported but missing some
  attributes), `aws_rds_cluster`, `aws_rds_cluster_instance`.
- **Auto Scaling / EC2** — `aws_launch_template`,
  `aws_autoscaling_group`, `aws_autoscaling_policy`,
  `aws_placement_group`. Common in any non-trivial compute setup.
- **Networking depth** — `aws_vpc_endpoint`, `aws_vpn_gateway`,
  `aws_dx_gateway`, `aws_route_table_association`,
  `aws_default_*` (the `default_*` resources need their own
  abstract-mapping pass).

Tranche 3 should prioritize EKS first (one cluster's worth of resources
unlocks ~5 modules in the smoke set), then RDS, then EC2 ASGs.

## How to re-run / extend

```powershell
# Default 10-repo smoke set
powershell -ExecutionPolicy Bypass -File scripts\validate-real-world.ps1

# Custom list (one repo URL per line, # comments OK)
echo "https://github.com/your-org/your-tf-repo" > my-repos.txt
powershell -ExecutionPolicy Bypass -File scripts\validate-real-world.ps1 -RepoList my-repos.txt
```

Output lands in `.validation-runs/<timestamp>/` (gitignored). Each run
produces:

- `summary.csv` — one row per repo with status, runtime, node/edge
  counts, warning counts, and the first warning text.
- `repos/<owner>/<name>/` — the cloned source (kept so you can
  re-grep without re-cloning).
- `logs/<owner>__<name>.{log,stdout,stderr}` — the full architex
  output triple per repo.

The script exits 0 unless ArchiteX panicked or returned a non-zero
status; warnings are tracked but never fail the run (the whole point of
this gate is to *find* them).
